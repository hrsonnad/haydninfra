Agent Roster

Five agents own different stages of the delivery pipeline. Click each to expand details.

Orchestrator Agent
Intake, decomposition, sequencing, and assignment
When Triggered
  • New vision/brain dump submitted
  • New epic needs story decomposition
Inputs
  • Raw vision notes or high-level idea
  • Constraints (timeline, design, tech, dependencies)
Outputs
  • Proposed epic name and goal statement
  • Sequenced story list with dependencies and sizing
  • Open questions requiring human decisions
  • Parallelization opportunities identified
Failure Modes
  • Stories too large or ambiguous — re-decompose
  • Missing constraints — surface open questions before proceeding
Spec Agent
Converts story intent into testable acceptance criteria
When Triggered
  • Story moves to Ready status in Linear
Inputs
  • Linear story with filled DoR template
Outputs
  • Clarified scope (in/out)
  • Binary pass/fail acceptance criteria
  • Edge cases and boundary conditions
  • Telemetry plan (event names, properties, verification)
  • Implementation steps (smallest safe sequence)
Failure Modes
  • Non-testable criteria — reject and revise
  • Scope creep — flag and defer to Orchestrator
Build Agent
Implementation planning and code execution
When Triggered
  • Story moves to In Progress status in Linear
Inputs
  • Spec Agent output (acceptance criteria, implementation steps)
  • Approved scope from story template
Outputs
  • File-level change plan before coding
  • Code changes completed with rationale
  • Tests added or updated
  • Manual QA steps (3–7 deterministic checks)
  • Remaining work or blockers surfaced
Failure Modes
  • Scope expansion — stop and surface the blocking decision
  • Hidden dependency — flag and pause for human input
QA Agent
Test matrix, acceptance validation, and regression checks
When Triggered
  • Parallel: drafts test matrix when story enters In Progress
  • Full execution: when story enters In Review
Inputs
  • Acceptance criteria from Spec Agent
  • Build report from Build Agent
Outputs
  • Pass/fail for each acceptance criterion
  • Regression check results
  • Edge case test outcomes
  • Telemetry verification evidence
  • Go/no-go recommendation with rationale
Failure Modes
  • Passing without evidence — flag unknowns explicitly
  • Missing telemetry — block ship until verified
Release Agent
Ship packet, rollback plan, and post-deploy verification
When Triggered
  • Story moves to Ready to Ship status in Linear
Inputs
  • QA go/no-go recommendation
  • Build report and change log
Outputs
  • Concise release note (user/business impact)
  • Risk summary at ship time
  • Explicit rollback steps
  • Post-ship verification checklist
  • Follow-up ticket recommendations
Failure Modes
  • Missing QA verification — do not ship
  • No rollback plan — block until provided

Story Lifecycle & Gates

Each status change has a gate. Stories cannot skip statuses or advance without meeting the gate requirements.

Backlog
Entry gate: None
Raw story ideas after Orchestrator decomposition. Not yet scoped or scheduled.
Ready
Gate: Definition of Ready
Scope is clear. Acceptance criteria are testable. Dependencies identified. Tracking plan defined. Spec Agent output posted as comment.
In Progress
Gate: Spec approved
Build Agent implements code. QA Agent drafts test matrix in parallel. Build report posted when complete.
In Review
Gate: Build complete
QA Agent executes full validation. Pass/fail each criterion. Verify telemetry events. Post go/no-go recommendation.
Ready to Ship
Gate: Definition of Done
All criteria pass. Tests complete. Telemetry verified. Release Agent produces ship packet with release note, risk summary, and rollback plan.
Shipped
Gate: Post-ship checks pass
Deployed and verified in production. Full artifact chain complete. Follow-up tickets created if needed.

Artifact Chain

Each stage produces a specific artifact. A complete chain is the success criteria for the workflow.

Ready → Spec brief (scope, ACs, edge cases, telemetry plan)
In Progress → Build report (files touched, tests, manual QA steps)
In Review → QA report (criterion-by-criterion pass/fail, telemetry verification)
Ready to Ship → Release packet (release note, risk summary, rollback plan)
Shipped → Post-ship verification + follow-up tickets

Parallelization

Most stages are serial, but one safe parallelization is built in: when a story enters In Progress, the Build Agent starts implementation while the QA Agent drafts its test matrix from the acceptance criteria. QA execution happens only after the build is complete.

Additional parallelization across stories is safe when they have no shared dependencies. The Orchestrator identifies these opportunities during decomposition.
